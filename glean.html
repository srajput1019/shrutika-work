Glean Agent Trust Harness
Forward-deployed · pre-production eval gate

Prove an agent is grounded, permission-safe, and finishes — before it ships.

A Glean Agent that lands in a customer's C-suite has to clear three bars at runtime: it stays grounded (cites a real, permitted source — never invents one), it respects permissions (never surfaces a doc the person asking can't see), and it runs to completion. This is the eval that gates the ship on those three axes. Flip a runtime guarantee off in the left rail and re-run — watch the harness catch the regression the way Glean Protect and the Enterprise Agent Development Lifecycle's "measure" phase are built to.

Axis 01 — Grounding

Cites a permitted source, or says nothing

Every claim binds to a retrieved doc above threshold, or the agent returns NULL_ANSWER. No source, no answer.

Axis 02 — Permission-respect

The asker's ACLs, enforced before the model

Glean's Enterprise Graph filters by permission before the LLM sees a snippet — the model never sees restricted data.

Axis 03 — Run-to-completion

Reaches a correct terminal state

The Agentic Engine finishes the multi-step flow — or correctly declines. No silent abandon, no partial write.

Why this maps to Glean, exactly

The three axes aren't generic — they're Glean's runtime thesis

Glean's whole security story is that trust is enforced by the architecture, not left to the model. The Enterprise Graph filters by permission before retrieval reaches the LLM; agents cite live, permitted sources; the Agentic Engine runs a real multi-step flow. This harness scores exactly those guarantees, so a green board means the build honors the thing Glean sells.

Grounding→Grounded citations — every agent output binds to a retrieved, permitted source or returns nothing
Permission→Enterprise Graph — per-document ACLs enforced before the model; "the LLM never sees restricted data"
Protect→AWARE · agent alignment — each write pre-scanned against the agent's purpose to block egress
Completion→Agentic Engine — reaches a terminal state; never emits a partial or abandoned run
The methodology behind it — and where it's already been proven

I've run this eval discipline on a real pipeline

I built and evaluated the Qlarc extraction engine — a working applied-AI pipeline — and wrote its harness (extract.py + eval.py). Anti-fabrication was architectural, not a prompt: the model's tool schema had no citation field, and every claim was fuzzy-matched against the exact page it cited — or dropped.
